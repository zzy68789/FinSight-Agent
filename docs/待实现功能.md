## P2：代码质量改造

基于对 `backend/` 的代码走查（130 个类、约 6985 行，无 God Class、无 TODO/FIXME、SSE 出口已集中、配置基本外置）。以下是最值得改的结构性问题，按子优先级排列。

### 高优先级

#### 任务生命周期模板去重

当前状态：`service/ResearchTaskService.java` 与 `financial/StockReportService.java` 各自实现了同一套任务骨架（`create → markRunning → try/catch{ markFailed + saveError + sseError }`），私有 `send()` 方法（`ResearchTaskService.java:76` vs `StockReportService.java:154`）几乎逐行相同。

建议方案：抽出 `TaskExecutionTemplate` / `TaskLifecycleRunner`，封装 create/running/failed/completed 与 send，两条链路只提供各自的工作流回调。这是全项目最大的重复面，任何对状态机、埋点、SSE 分发的调整都要改两处且易漏改。

#### Repository 层集成测试

当前状态：`repository/` 7 个类 + `financial/FinancialSnapshotRepository.java` 全是手写 SQL（`JdbcClient`/`JdbcTemplate`），但唯一的 `@SpringBootTest` 排除了数据源、把所有 repository mock 掉。列名、`LIKE` 拼接、`RowMapper` 字段映射、JSON 序列化没有任何自动化验证。

建议方案：引入 H2 / Testcontainers 的 `@JdbcTest` 级集成测试，至少覆盖 `ResearchTaskRepository`、`ReportRepository`、`FinancialSnapshotRepository` 的增删查。

#### StockReportService 编排层单测

当前状态：`financial/StockReportService.java`（179 行，含 85 行的 `runWorkflow`）完全没有单元测试，而它串起的 `StockReportWorkflow`、各 reviewer、writer 都有测试。其中"最多重试 3 次直到 review + compliance 双 PASS"的循环（`:107-124`）是财务链路唯一有分支/循环的编排逻辑，回归风险最高却零覆盖。

建议方案：补 `StockReportService` 单测，覆盖重试循环、双 PASS 判定和失败收尾分支。

#### TuShare HTTP 超时防线程耗尽

当前状态：`financial/TushareMarketDataProvider.java:40` 构造器接收 `timeout` 参数却从未使用，`RestClient` 未设任何读超时。外部 HTTP 挂起会占满 `financialProviderExecutor`（仅 6 线程，`AsyncExecutionConfig.java:21`），形成隐性资源耗尽。

建议方案：给 `RestClient` 配置连接和读超时，用上已注入的 `timeout` 值。这是健壮性缺陷，不是纯清理。

### 中优先级

#### FinancialEvidenceItem 工厂方法

当前状态：12 参数的 `new FinancialEvidenceItem(...)` 构造器散落在 8 处 provider；每个 provider 各写一份 `dataMissing(...)` 兜底（confidence=0、`DATA_MISSING`）。12 个位置参数无编译期字段名保护，极易错位。

建议方案：集中到工厂方法 / builder，如 `FinancialEvidenceItem.dataMissing(...)`，并用 builder 替代长构造器。

#### InvestmentReportWriter 章节渲染去重

当前状态：`financial/InvestmentReportWriter.java` 主报告内联的"引用与数据快照 + 指标公式 + 风险明细"（`:71-105`）与已抽出的 `appendCitationSection`（`:201-242`）是同一段逻辑的两份拷贝；`write` 97 行、`writeEtfReport` 60 行。

建议方案：主报告改为调用 `appendCitationSection`，各 section 拆成小方法。

#### 魔法数字外置为常量/配置（部分完成）

已完成：通用调研链路的检索 topK、相关度阈值、工具调用轮次已外置为 `AgentProperties`（`drai.agent.*`），`ResearcherNode` 不再有硬编码 5/3/0.2。

待完成：金融链路 `StockReportService.java:107` 裸 `attempt < 3` 与 `ResearchRoutePolicy.MAX_REVISIONS` 未统一；证据截断长度不一致（320 vs 300）仍待收口。

#### AdminService 职责拆分

当前状态：`service/AdminService.java` 有双构造器（无参 + `@Autowired` 十参），且业务 service 直接持有 `JdbcTemplate`/`StringRedisTemplate` 内联 `SELECT 1`/`ping` 做健康检查（`:137-168`），基础设施探测与业务混杂。

建议方案：健康检查抽为独立 `HealthIndicator`；去掉为测试留的双构造器后门。

### 低优先级

#### 补齐 agent 节点与提示词测试

当前状态：`agent/node/` 仅 Researcher/Reviewer/Router 有测试，`PlannerNode`/`WriterNode`/`RefinerNode` 无；`llm/PromptTemplates`、`llm/LlmFallbacks`（按 ModelType 分支的降级文案）也无测试。

建议方案：补三个节点单测使 agent 五节点覆盖完整；对 `LlmFallbacks` 加快照测试，提示词变更有回归保护。

#### 收敛异常吞没与 SSE 超时

当前状态：`AdminService.java:148/165` 的 `catch(Exception)` 把异常压成 "DOWN" 不记日志；`ChatController.java:26` `new SseEmitter(0L)` 永不超时，客户端异常断开时 emitter 可能长期悬挂。（注：`TaskRuntimeStateService` 的 `catch(RuntimeException ignored)` 是有注释的 Redis 降级，属合理设计，无需改。）

建议方案：健康检查失败至少 `log.debug` 记录原因；给 SSE emitter 设合理超时或依赖容器超时。

## P3：金融垂直能力增强
