## FinSight 命名与数据隔离

- Java 根包已从旧标识迁移为 `com.zzy.finsight`，Spring Boot 启动类统一为 `FinSightApplication`，Maven artifactId 和应用名统一为 `finsight-backend`。
- 配置命名空间统一为 `finsight.*`；只有 LLM、Tavily、TuShare 三类 API Key 使用 `API_KEY`、`TAVILY_API_KEY`、`TUSHARE_API_KEY` 环境变量，其余配置直接写在 `application.yml`；`ConfigurationPlaceholderTest` 同时防止引入多余占位符和回退到全局 `OPENAI_API_KEY`。
- MySQL 默认连接全新 `finsight` 库，不复用旧库；Redis 默认使用逻辑库 7 和 `finsight:*` key，ChromaDB 使用独立 `finsight_docs` collection。
- 前端登录态存储键已改为 `finsight_token`，页面标题和产品文案统一使用 FinSight。

## 证券代码研究报告 MVP

- 新增 `POST /api/stock-reports` SSE 入口，支持 `ticker`、`thread_id`、`report_period`、`search_mode`。
- 证券代码解析支持普通 A 股 `6xxxxx -> .SH`、`0xxxxx / 2xxxxx / 3xxxxx -> .SZ`，以及常见 ETF `5xxxxx -> .SH`、`15xxxx / 16xxxx / 18xxxx -> .SZ`。
- 不支持的代码会通过 SSE `error` 事件返回明确业务原因，不再在前端泛化为“网络异常”；当前仍不支持 B 股、债券或北交所标的。
- 新增金融数据模型：`StockSubject`、`FinancialEvidenceItem`、`FinancialSnapshot`、`FinancialMetricResult`。
- 新增 `FinancialDataProvider` 扩展点，已实现 `UploadedReportProvider`、`PublicMarketDataProvider` 和 `TushareMarketDataProvider`。
- 新增 `AShareCompanyDirectory` 和 `AShareMasterDataProvider`，为 10 个默认评测样例股票补齐公司名称、行业和 `LOCAL_CONTEXT` 证据。
- LLM 只有 API Key 使用用户指定的 `API_KEY`，地址、模型、超时和重试参数直接维护在 `application.yml`；不读取全局 `OPENAI_API_KEY`，避免影响 `cc-switch` 等全局配置。
- 新增 TuShare Pro 授权数据源配置，只有 token 使用 `TUSHARE_API_KEY`，启用开关、地址和超时直接维护在 `application.yml`；未配置 token 时不调用外部接口。
- `TushareMarketDataProvider` 已能把普通股票的 TuShare `income`、`balancesheet`、`cashflow`、`daily_basic` 响应映射为营业收入、营业成本、归母净利润、资产负债、经营现金流、市盈率、市净率和总市值证据；财报行按报告期、标准合并报表、`update_flag` 和公告日期选择唯一版本，上年同期严格按本期日期减一年匹配；ETF 标的优先走 `fund_daily` 并映射收盘价、涨跌幅和成交额证据。
- `FinancialEvidenceValidator` 在快照进入指标引擎前标记重复报告期、上年同期错配、未来日期、资产负债异常和低质量网页正文；公开新闻按 URL 去重，并识别登录页、导航页、行情 SEO 占位内容和链接密度异常，任意非空 `issueCode` 均不再计入有效证据。
- `FinancialSnapshotBuilder` 已支持 provider fan-out 并行采集，并为每个数据源记录状态、耗时、证据数量和失败信息。
- 股票报告任务和金融 provider fan-out 使用 Spring 管理的有界执行器，线程数通过 `finsight.async.financial-provider-threads` 配置。
- 公开数据失败时不会阻断流程，会写入 `DATA_MISSING` 证据。
- 新增 `FinancialMetricEngine`，普通股票用 Java `BigDecimal` 计算营收同比、毛利率、净利率、ROE、资产负债率、经营现金流 / 净利润；ROE 新口径使用年初与期末归母权益的平均值，上传报告直接披露的平均净资产仅作为兼容路径；ETF 不套用公司财务指标，只输出 ETF 收盘价、涨跌幅和成交额等行情指标。
- 新增 `MetricDefinitionCatalog`，集中声明指标公式、`formulaVersion` 和输入依赖；`stock_metric_result.formula_version` 保存实际使用的公式版本。
- 缺少分子或分母时返回 `MISSING_INPUT`，报告中显示“数据缺失”。
- `FinancialRiskScorer` 按基本面 30%、技术面 25%、情绪面 20%、消息面 15%、市场环境 10% 输出五维风险评分；一季度、半年度和前三季度 ROE 明确按未年化阶段性指标处理，不再套用全年 ROE 阈值。
- `FinancialComplianceReviewer` 检查免责声明、保证收益、内幕信息、直接荐股/仓位指令、无审慎边界的股价方向表达和引用审查失败，并输出结构化 `issues` 与合规分数。
- `FinancialEvaluator` 已升级为 `financial-evaluation-v4-layered-gates`：所有股票和 ETF 都执行数字一致性、正文引用、报告期、方向性判断和低质量引用硬门禁，不再因未命中 10 个默认代码而跳过；来源质量与有效证据率降为 `ADVISORY/WARN`，不会因临时数据源缺失单独阻断报告。
- 新增独立金融工作流：`StockResolve -> DataSnapshot -> MetricEngine -> RiskAssessment -> EvidenceCollect -> InvestmentWriter -> CitationReviewer + ComplianceReviewer -> Evaluation`。
- `InvestmentReportWriter` 已接入 `LlmClient` SMART 模型：LLM 接收八章节叙述草稿和最多 18 条有效紧凑证据索引，正文财务、估值、新闻和风险结论要求就近标注 `[E#]`；固定免责声明和完整引用附录仍由 Java 确定性补齐。
- Writer 能识别一季度、半年度、前三季度和年度口径，阶段性 ROE 固定标注“未年化”；估值章节优先使用结构化 `PE_TTM`、`PB` 和总市值，结论只列出真实缺失项，不再输出泛化缺失模板。
- LLM 输出通过 `finsight.llm.max-output-tokens` 限制，当前为 4096；Writer 版本升级为 `investment-report-writer-v5-cited-period-aware`，避免复用旧生成上下文缓存。
- 报告保存 `generation-mode: llm|template-fallback` 与结构化 `fallback-reason` 隐藏标记；超时、结构非法、未配置、空响应和调用失败分别归类为 `LLM_TIMEOUT`、`LLM_INVALID_STRUCTURE`、`LLM_NOT_CONFIGURED`、`LLM_EMPTY_RESPONSE`、`LLM_CALL_FAILED`。
- 普通股票报告和 ETF 报告均固定八章节，并在底部输出“引用与数据快照”和指标公式；ETF 报告使用基金概况、跟踪标的、行情、流动性、净值/持仓缺口等基金口径章节。
- `CitationReviewer` v3 只在正文范围检查指标值和 `[E#]`，要求指标所在句引用对应原始字段，拒绝不存在或无效的证据编号、阶段性 ROE 口径错误、静态 PE 无结构化证据和跨网页行情拼接；最终通过条件为引用审查与合规审查同时 PASS。
- 报告统一标注“仅作研究辅助，不构成投资建议”。

## 工作流可靠性与报告复现

- `StockReportService` 已拆分为异步入口和可同步测试的 `StockReportRunner`；SSE 断开只关闭进度通道，不再把后台任务误标为失败。
- `research_task` 已记录 `stage`、`attempt_count`、`request_payload`、`heartbeat_at`、数据库租约和最近错误；`StockReportRecoveryScheduler` 每 60 秒扫描超过 5 分钟无心跳的股票报告任务，最多执行 3 次。
- 新增 `POST /api/stock-reports/{taskId}/retry`，只允许任务所属用户重试未耗尽次数的 `FAILED` 任务。
- 恢复执行会优先复用当前任务已保存的 snapshot、metric 和 report，避免重复采集与重复落库。
- `FinancialReportFingerprinter` 对稳定业务字段生成 `data_snapshot_hash`，并结合公式目录、Writer、Citation、Compliance、Evaluation 规则版本生成 `generation_context_hash`。
- 同一用户下只有 `generation_context_hash` 相同、评审为 `PASS` 且未删除的报告允许复用；新任务仍创建报告版本并记录 `reused_from_report_id`，不跨用户共享缓存。

- 上传报告检索 trace 随 `FinancialSnapshot` 持久化，并通过 `agent_step_log` 绑定到具体任务；步骤日志新增 `attempt_no` 和 `duration_ms`。

- `FinancialEvaluator` 在原 claim/citation/numeric/keypoint 指标上增加 `evidence_effective_rate`；`unsupported_claim_rate` 仍是规则型无依据断言检查，不作为完整幻觉检测宣传。
- 新增 `GET /api/stock-reports/{taskId}/trace`，聚合任务阶段、报告版本、双哈希、缓存来源、证据统计、检索 trace、步骤耗时和 Reviewer 摘要。

- 已接入 Spring Boot Actuator、Micrometer Prometheus；暴露 `/actuator/health`、`/actuator/prometheus`，记录任务结果、阶段耗时、缓存命中和恢复结果。

## 复用的通用基础设施

金融链路复用以下基础能力（这些能力保留在本仓库，但不再对外暴露通用 deep-research 入口）：

- RAG：PDF 上传、文本切片、embedding、BM25 + ChromaDB 向量混合检索（`rag` 包），用于上传年报/季报作为证据来源。
- 联网搜索：`SearchSource` 抽象 + Tavily Search + `TavilyExtractClient` 批量正文提取；最多提取 3 个候选 URL，单次服务端超时 10 秒，只对网络错误和 5xx 最多重试 1 次，失败后使用合格搜索摘要或写入缺失证据。
- 持久化：任务、Agent 步骤日志、报告版本、checkpoint 和金融快照统一通过 MyBatis Mapper 持久化；Mapper 接口位于 `mapper` 包，SQL 位于 `resources/mapper`。
- 报告库：报告列表、版本、导出（PDF/Word/Markdown）、收藏、软删除、追加入 RAG（`ReportService` / `ReportExportService`）。
- 用户与后台：注册/登录、Bearer Token 鉴权、`owner_id` 数据隔离、Admin 后台与审计日志。
- LLM 基础设施：`LlmConfig` 提供 fast/smart/judge 三档 ChatModel；`LlmClient.generateWithMetadata` 在保留原 `generate` 调用的同时返回模型名、输入/输出 Token、结束原因和耗时，`RetryingLlmClient` 保留完整元数据重试结果。
- SSE 实时推送：`SseService` 统一出口。

## 评测体系

- `backend/src/test/resources/evaluation/dataset-v1.json` 包含 10 个 A 股、2 个 ETF 冻结正例，8 个数字/引用/报告期/方向性/合规对抗负例，以及 24 个带相关性等级的检索问题；主回归不访问 TuShare、Tavily、ChromaDB 或真实 LLM。
- `FinancialEvaluationRunner` 一次执行规则期望校验、默认 `0.01` 绝对误差校验和真实 `HybridRagRetriever` 检索，生成 Recall@1/3/5、Precision@1/3/5、MRR、NDCG@1/3/5。
- `baseline-v1.json` 保存显式评测基线；`Recall@3 >= 0.90`、`MRR >= 0.80`、`NDCG@3 >= 0.85`，核心检索指标相对基线下降超过 `0.05` 时阻断。
- 每次运行在 `backend/target/evaluation/<runId>/` 生成 `results.json` 和 `summary.md`；只有设置 `finsight.eval.update-baseline=true` 才会更新版本库基线。
- `LlmJudgeEvaluator` 只依据冻结证据输出 relevance、faithfulness、completeness、research_helpfulness 四维结构化评分；非法 JSON 明确记为 `ERROR`，单项低于 `0.80` 只产生非阻断告警。
- `LlmJudgeEvaluationTest` 与 `LiveProviderEvaluationTest` 默认关闭；显式开启但缺少三类 Key 时写出 `SKIPPED` 报告，不伪装成通过。两个 GitHub Actions 工作流分别负责 PR 确定性门禁和定时/手动外部评测。

## MyBatis 与 Spring MVC 分层

- 后端持久层已从手写 JDBC Repository 迁移到 MyBatis 3.0.4，调用关系统一为 `controller -> service -> mapper`；Service 保留业务编排，Mapper 只负责数据库访问。
- 新增 9 个 Mapper 接口和对应 XML，覆盖用户、任务、步骤日志、报告、checkpoint、金融快照、后台管理、审计日志及数据库探活。
- JSON 列通过 `infrastructure/persistence/mybatis/typehandler` 下的三个 TypeHandler 统一序列化，不再与 Mapper 接口混放。
- 原 `financial` 大包已删除：金融领域对象位于 `domain/stock`，接口 DTO 位于 `dto`，指标、采集、审查和股票报告编排分别位于 `component/analysis`、`component/marketdata`、`component/review`、`component/workflow`。
- `service` 根目录只保留 Controller 面向的服务接口，具体实现统一位于 `service/impl`；原位于 `rag` 包的业务入口已迁移为 `service.RagService` 与 `service.impl.RagServiceImpl`，`rag` 仅保留检索器、向量存储、切片器和 RAG 数据模型。
- 原 `repository` 包已删除；`MapperXmlConfigurationTest` 会解析全部 Mapper XML，并检查核心 SQL statement 是否完成注册。
- `ArchitectureLayeringTest` 已加入 `RagService` 契约与 `RagServiceImpl` 分层断言，并继续校验业务组件、MyBatis TypeHandler 和中文 Javadoc 边界。
