# FinSight-Agent 最终优化计划

> 实施状态（2026-07-10）：核心改造已落地并通过后端 90 个测试和前端生产构建。Flyway 已接管迁移，但本机 Docker/MySQL 不可用，真实 MySQL 迁移集成验证仍保留为后续项。

## Summary

最终结论：Claude 的方向基本正确，但需要补回“工作流恢复”，并避免重复建设。当前项目已经有报告版本、引用评测、数值一致性评测和确定性 Writer；真正需要补的是编排测试、运行 trace、数据指纹、公式审计、轻量恢复和可视化。

目标按“实习面试主项目”排序：先做可信、可测、可演示的能力，再做监控与数据库治理。

## Step 1：补齐核心编排测试与生命周期正确性（P0）

- 从 [StockReportService.java](D:/Code/FinSight-Agent/backend/src/main/java/com/zzy/finsight/financial/StockReportService.java) 抽出可同步测试的 `StockReportRunner`，原 Service 只负责线程池提交和 SSE 连接。
- 新增 `StockReportServiceTest`，覆盖首轮评审失败后重写通过、三轮仍失败、合规失败、评测失败、组件异常后任务落为 `FAILED`。
- SSE 发送失败只记录客户端断开，不得中断后台报告生成或把任务标记为失败。
- 固定最大 Writer 尝试次数为 3；只有 Citation、Compliance、Evaluation 全部通过，最终状态才为 `PASS`。

## Step 2：建设 RAG 与报告全链路 Trace（P0）

- 为 `HybridRagRetriever` 新增 `retrieveWithTrace()`，返回文档结果及关键词分数、向量分数、融合分数、排名、候选数、阈值过滤数和耗时；保留原 `retrieve()` 兼容现有调用。
- 扩展 `agent_step_log`，增加 `attempt_no`、`duration_ms` 字段；继续使用 `output_snapshot` 保存步骤特有指标，不新增功能重叠的 `stock_report_trace` 表。
- 统一记录 snapshot、metric、risk、retrieval、writer、citation reviewer、compliance reviewer、evaluation 的状态、耗时和尝试次数。
- `FinancialEvaluationService` 保留现有六项评分，新增证据有效率；检索接受率作为运行指标展示，不混入报告质量总分。
- 现有 `unsupported_claim_rate` 明确定义为“规则型无依据断言检测”，不宣传为完整幻觉检测。

## Step 3：增加公式版本与报告数据指纹（P0）

- 新增 `MetricDefinitionCatalog`，集中声明指标名称、公式文本、`formulaVersion` 和输入依赖；当前六个股票指标和三个 ETF 指标不建设通用 DAG 引擎。
- `FinancialMetricResult` 和 `stock_metric_result` 增加 `formula_version`，所有新增字段带中文 `COMMENT`。
- `stock_analysis_snapshot` 增加 `data_snapshot_hash CHAR(64)`：对排序后的证券信息、报告期、有效证据值、数据时间、缺失状态进行 SHA-256；排除任务 ID、线程 ID、抓取耗时等波动字段。
- `report` 保留现有 `version`，新增 `snapshot_id`、`data_snapshot_hash`、`generation_context_hash`、`reused_from_report_id`。
- `generation_context_hash` 由数据哈希、公式目录版本、Writer 版本和 Reviewer 规则版本共同生成，代码或规则升级后自动失效旧缓存。
- 仅复用同一用户、同一证券、相同 context hash、评审为 `PASS` 且未删除的报告；命中后仍创建当前任务的报告记录，保证回放和租户隔离。

## Step 4：实现轻量可恢复工作流（P1）

- `research_task` 增加 `stage`、`attempt_count`、`request_payload`、`last_error`、`heartbeat_at`、`lease_owner`、`lease_until`。
- 状态固定为 `CREATED -> RUNNING -> RETRYING -> RUNNING -> COMPLETED/FAILED`，最大工作流尝试次数为 3。
- 每 60 秒扫描一次超过 5 分钟未更新的 `RUNNING` 任务；有剩余次数则进入 `RETRYING`，否则进入 `FAILED`。
- snapshot、metrics、report 写入按 `task_id` 做幂等检查；恢复时从最后一个已持久化阶段继续，避免重复数据。
- 新增 `POST /api/stock-reports/{taskId}/retry`，校验任务归属，只允许重试 `FAILED` 任务。
- 当前单实例使用数据库条件更新抢占任务；Redis fencing-token 仅在未来多实例部署时增加，不进入本轮。

## Step 5：增加可信度面板与运行监控（P1）

- 新增 `GET /api/stock-reports/{taskId}/trace`，返回任务阶段、尝试次数、报告版本、两个哈希、缓存来源、证据统计、检索 trace、步骤耗时和三类评审结果。
- 前端任务/报告页将原始回放 JSON 改为可扫描的可信度区域，展示证据有效率、检索来源、重写次数、评审状态、快照哈希和缓存命中情况。
- 引入 Actuator 与 Micrometer Prometheus，记录阶段耗时、任务结果、报告缓存命中和恢复次数，暴露 `/actuator/health` 与 `/actuator/prometheus`。
- 保持 `/api/stock-reports` 请求体、SSE 步骤名和现有字段兼容。

## Step 6：数据库迁移治理（P2）

- 前述字段先通过 `upgrade-*.sql` 落地，避免同时改业务和迁移框架。
- 功能稳定后引入 Flyway，将初始化脚本和历史升级脚本整理为顺序迁移。
- 已存在的 MySQL 数据库使用 baseline 接管；空库必须能从迁移脚本完整初始化。
- 增加 MySQL/Testcontainers 仓储集成测试，验证哈希缓存、公式版本、任务抢占和恢复查询。

## 验证

- 后端单测覆盖重写循环、双 Reviewer、异常落库、SSE 断开、哈希稳定性、缓存失效、租户隔离和恢复次数上限。
- RAG 测试覆盖关键词命中、向量命中、重复结果融合、阈值过滤及 trace 分数。
- 执行 `mvn.cmd test` 和 `npm.cmd run build`。
- 手工验证相同数据第二次生成命中缓存，公式或 Writer 版本变化后不命中缓存。
- 手工中断 SSE 和模拟进程重启，确认任务不会因客户端断开失败，过期任务能够恢复或明确终止。

## 明确不做

- 不引入 Python 边车、RabbitMQ、Elasticsearch、MinIO 或 PostgreSQL/pgvector。
- 不宣称或照搬外部项目未实现的多空辩论 Agent、LangGraph 编排和 token 统计。
- 暂不把指标计算改造成通用 DAG；出现指标依赖指标的真实需求后再升级。
- 暂不接入 LLM Writer；当前确定性 Writer 已直接使用结构化指标和风险结果。未来接入时必须保留模板降级与双 Reviewer。
